Fabric chaincode lifecycle

Fabric 链码生命周期

什么是链码？

链码是一段用 Go、Node.js 或者 Java 写的程序，该程序实现了规定接口。链码运行在一个受保护的Docker容器当中，与背书节点的运行互相隔离。链码可通过客户端提交的交易对账本状态初始化并进行管理。

链码通常用来处理网络中成员一致认可的业务逻辑，所以它被称为“智能合约”。通过链码创建的账本更新操作只能在该链码中被调用，并且不能被另一个链码直接访问。然而在同一个网络中，在被授予适当权限的情况下，链码可以调用另一个链码来访问它的状态。

在这个主题下，我们将会以一个区块链网络操作者，而不是应用开发者的视角来探索链码。链码操作者可以用本文作为指南，在网络上使用 Fabric 链码生命周期来部署和管理链码。

部署一个链码

Fabric 链码生命周期是一个过程，它允许多个组织在使用一个链码之前，就如何操作达成共识。网络操作者可以使用 Fabric 生命周期来展开以下的任务：

  • 安装和定义链码
  • 升级链码
  • 部署场景
  • 迁移到新的 Fabric 生命周期

你可以通过创建一个版本为 V2_0通道来使用 Fabric 链码生命周期。在版本为 V2_0 的通道上，不能使用旧的生命周期来安装、初始化或升级链码。但是，在启动 V2_0 后，仍然可以调用之前生命周期模式安装的链码。

安装以及定义链码

Fabric 链码生命周期需要定义链码的参数（链码定义），比如说名称、版本以及链码背书策略，参数的定义需要得到组织的同意。通道成员通过以下四步达成共识，并不是通道上的每个组织都需要完成以下所有步骤。

1、打包链码： 这一步可以被一个或者每一个组织完成。

2、在对等节点上安装链码： 每一个用链码的组织都需要完成这一步。

3、批准链码定义： 使用链码的每一个组织都需要完成这一步。在链码在能够通道上运行之前，链码定义需要被足够多的（默认为大多数）组织批准，来满足通道的生命周期背书策略。

4、把链码定义提交到链上： 一旦通道上已经有所需数量的组织同意该链码定义，就需要一个组织提交交易。提交者首先从已批准组织中的对等节点收集背书，然后通过提交交易来提交链码声明。

步骤一：打包智能合约

链码需要被打包进一个 tar 文件，然后才能被安装到各组织的对等节点。你可以使用 Fabric 对等节点的二进制可执行文件、Node Fabric SDK 或者第三方工具（例如，GNU tar ）来打包链码。在创建链码包的时候，需要提交一个链码包标签，作为一个简明易读的包描述。

如果你使用第三方工具来打包链码，需要打包成以下格式。Fabric 对等节点 的二进制可执行文件和 Fabric SDK 将会自动以这个格式来创建文件。

  • 链码需要被打包进一个文件扩展名为 .tar.gz 的 tar 文件。

  • tar 文件需要包含两个文件（没有目录）：“metadata.json”和另一个包含了链码文件的 tar 文件“code.tar.gz”。

  • “metadata.json”包含 JSON 文件，在这个 JSON 文件中指明链码语言、代码路径、以及包标签。下面是一个JSON 文件的例子：

    {"Path":"fabric-samples/chaincode/fabcar/go","Type":"golang","Label":"fabcarv1"}

链码分别被 Org1 和 Org2 打包。为了使用名字和版本来识别包，两个组织都使用 MYCC_1 作为包标签。对于多个组织来说没有必要使用相同的包标签。

步骤二：在对等节点上安装链码

你需要在每个要执行交易和对交易背书的对等节点上安装链码包。无论使用 CLI 还是 SDK，都需要以 Peer Administrator （对等节点所在组织的管理员） 的身份来完成这步。链码安装后，对等节点会构建链码，如果链码有问题，会构建失败，并返回构建错误。建议每个组织只打包一次链码，然后在他们的对等节点上安装相同的包。如果某个通道希望确保每个组织都运行同样的链码，可以让某一个组织可以打包链码，然后通过其他方式（不通过链上）把打包的链码发送给其他通道成员.

链码成功安装后会返回链码包标识符，它是包标签和包哈希值的结合。这个包标识符用来关联对等节点上安装的链码包和被批准的链码。保存这个标识符，下一步操作会用到。你也可以使用 Peer CLI 工具查询安装在对等节点上的包，以此查看包标识符。

Org1 和 Org2 的管理员安装链码 MYCC_1 到加入通道的对等节点上。安装链码包构建了链码并且创建了包标识符 MYCC_1:hash。

步骤三：为你的组织批准链码定义

链码的管理通过链码定义实现。当通道成员批准一个链码定义，就表示该成员接受链码参数。通道成员要在链在通道上使用之前就链码达成一致（同意链码运行在此通道上）。链码定义包含了以下需要持续在组织之间保持一致的参数：

  • 名称： 应用调用链码时使用的名称。

  • 版本： 一个版本号或者和给定链码包关联的值。如果要升级链码二进制文件（打包后的链码文件），链码版本也需要改变。

  • 序列号： 链码被定义的次数。这个值是一个整数，并且被用来追踪链码的更新次数。例如当你第一次安装并且同意一个链码定义，这个序列号会是1。当你下一次更新链码，序列号会是2。

  • 背书策略： 哪些组织需要执行并且验证交易输出。背书策略可以表达为传递给 CLI 工具的字符串，也可以在通道配置中引用策略。。默认情况下，背书策略设置为 Channel/Application/Endorsement，要求通道中大多数组织认可该交易。

  • 集合配置（私有数据集合配置）： 定义和链码相关的私有数据集合的文件路径。了解更多关于私有数据集合的信息，查看 私有数据架构参考。

  • ESCC/VSCC 插件： 这个链码使用的自定义背书策略或者有效插件的名称。

  • 初始化： 如果你使用 Fabric Chaincode Shim API 提供的低级别的 API，你的链码需要包含用来初始化链码的 Init 方法。 链码接口需要Init 方法，但是应用不是必须要调用它。 当你批准一个链码定义时，你可以指定是否必须在调用其他方法之前执行 Init 方法。如果你指定需要 Init方法，Fabric 会在调用其他方法之前调用Init 方法，并且只会被调用一次。 执行 Init 方法可以让初始化链码运行逻辑，例如设置一些初始状态。假定链码定义增加版本号需要调用 Init 方法，那么每次更新链码版本，都需要调用 Init 来初始化链码。

    如果使用 Fabric peer CLI 工具，当你同意提交链码定义时候，你可以使用 --init-required 来表示 Init 方法必须被调用来初始化新的链码版本。用 Fabric peer CLI 工具调用 Init 方法时，需要使用 peer chaincode invoke 指令并传递 --isInit 变量。

    如果使用 Fabric 合约 API，不需要把 Init 方法包含进你的链码。但是，你仍然可以使用 --init-required 发起调用Init 方法来初始化链码的请求。如果你使用 --init-required ，需要传 --isInit 变量给链码，以便实现每次初始化链码时，链码的版本号增加。你可以用链码中的任何方法结合传 --isInit 变量来初始化链码。

    链码定义也包括 包标识符。这是每个想要使用链码的组织所必需的参数。对于每个组织来说，包 ID 不需要一致。一个组织可以在不安装链码包或者链码定义中不包含标识符的情况下，批准链码定义。

每个希望使用链码的通道成员需要批准其组织的链码定义。这个批准操作需要提交给排序服务，在此之后会分发给所有的对等节点。这个批准操作需要被你的 组织管理员 提交。在批准交易被成功提交后，已批准的定义将存储一个集合中，所有的对等节点都可访问这个集合。因此，即使你有多个对等节点，也只需要为你的组织批准一次链码定义。

Org1 和 Org2 的组织管理员为他们的组织批准 MYCC 链码定义。链码定义的其他字段包括链码名称、版本号和背书策略。因为两个组织都会用链码来背书交易，所以两个组织批准的定义中都需要包含 packageID。

步骤四：提交链码定义到通道

一旦足够多的通道成员同意一个链码定义，某个组织就能够把该链码定义提交到通道上。在使用 Peer CLI 工具将链码定义提交到通道之前，你可以用 checkcommitreadiness 命令，根据哪个通道成员已经批准了该定义，来检查是否能成功提交链码定义（根据通道成员同意的状况，来判断提交是否可能成功）。提交交易请求首先发送给通道成员的对等节点，对等节点会查询其组织是否批准了链码定义，如果批准了，则认可该定义。然后将交易提交给排序节点，排序节点会把链码定义提交给通道。提交链码定义交易需要以 组织管理员 的身份来进行。

在成功地将链码定义提交到通道之前，需要批准链码定义的组织数量通过 Channel/Application/LifecycleEndorsement 策略控制。默认情况下，这个策略需要通道中大多数的组织来给交易背书。生命周期背书策略不同于链码背书策略。例如，尽管一个链码背书策略只需要一个或两个组织的签名，但根据默认策略仍需要大多数的组织签名。在提交一个通道定义时，仍然需要足够多的组织，以确保满足生命周期背书策略。你可以在策略概念主题中，了解到更多关于 Fabric 链码生命周期策略的信息。

你也可以设置 Channel/Application/LifecycleEndorsement 策略作为签名策略，并且明确出通道上可以批准链码定义的组织集合。这样你可以创建一个通道，其中大多数组织作为链码管理者，也一起管理通道业务逻辑。如果你的通道有大量的 Idemix（身份混合）组织，你也可以用一个签名策略（只需要一个签名），因为这些Idemix组织不能批准链码定义或者为链码背书，还有可能阻碍通道达成大多数成员同意的结果。

Org1 或 Org2 的一个组织管理员提交链码定义到通道。通道上的链码定义不包含 packageID。

一个组织在不安装链码包的条件下能够批准链码定义。如果一个组织不需要使用链码，他们可以在没有包身份的情况下批准一个链码定义，来确保生命周期背书策略被满足。

在链码定义已经提交到通道上后，链码容器会在所有安装链码的对等节点上启动，来允许通道成员开始使用链码。启动链码容器可能会花费几分钟的时间。你可以用链码定义来要求调用 Init 方法初始化链码，也可以不要求调用该方法。如果需要调用Init 方法，那么链码必须第一个调用该方法。Init 方法的调用服从于链码的背书策略。

一旦 MYCC 链码在通道上定义，Org1 和 Org2 就能开始使用链码。每个对等节点在第一次调用时会启动该对等节点上的链码容器。

升级链码

你可以用相同的 Fabric 生命周期流程（和部署链码的生命周期相同）来升级链码。你可以升级链码二进制文件或者只升级链码策略。升级链码的步骤如下：

1、重新打包链码： 如果要升级二进制文件，只需要完成这一步就可以。

Org1 和 Org2 升级链码二进制文件并且重新打包链码。两个组织都用不同的包标签。

2、在对等节点上安装新的链码包： 再次复述，如果你要升级链码二进制文件，那么你只需要完成这一步。安装新的链码包会生成一个新的packageID，你需要把它传给新的链码定义。如果链码二进制文件升级了，也需要修改链码版本，生命周期进程使用链码版本来跟踪链码二进制文件是否已经升级。

Org1和Org2在他们的 对等节点上安装新链码包。安装会生成一个新的 packageID。

3、 批准一个新的链码定义： 如果你正在升级链码二进制文件，你需要在链码定义中升级链码版本和packageID。也可以在不打包你的链码二进制文件情况下，升级链码背书策略，通道成员只需要在新的策略中批准一个定义，新的定义需要把 sequence（序列号） 变量的值加1。

Org1 或者 Org2 的组织管理员为他们的组织批准新的链码定义。新的定义引用新的 packageID 并且改变链码版本。因为这是第一次升级链码，所以序列号从1增加到2。

4、 提交定义到通道： 当有足够多的通道成员批准新的链码定义时，某个组织可以提交新的定义来升级通道上的链码定义。生命周期过程中不存在单独的升级命令。

Org1 或者 Org2 的的管理员提交新的链码定义到通道上。

你提交链码定义后，会启动一个新的链码容器，这个链码容器中包含升级后的链码二进制文件。如果在链码定义中请求执行 Init 方法，需要在成功提交新的定义后再次调用 Init 方法来初始化升级过的链码。如果在链码升级时没有改变链码的版本号，那么链码容器不会更新，也不需要调用Init 方法。

一旦新的定义提交到链上，每个对等节点会自动启动新的链码容器。

Fabric 链码生命周期在链码定义中使用 sequence（序列号） 来跟踪升级。所有通道成员都需要将序列号加1，并批准一个新的定义来升级链码。版本参数用来追踪链码二进制文件，并且只有当你升级链码二进制时需要改变。

部署场景

以下示例说明了如何使用 Fabric 链码生命周期来管理通道和链码。

加入通道

一个新的组织可以在链码已经定义的情况下加入通道，并且新的组织在安装完链码包、批准已经提交到通道的链码定义之后，可以开始使用链码。

Org3 加入通道并且批准之前已经被 Org1 和 Org2 提交到通道的链码定义。

在批准链码定义后，新的组织会它们的对等节点安装好链码包之后，开始使用链码。定义不需要被再次提交。如果使用默认的背书策略，那么需要大多数通道成员背书，背书策略会自动更新，把新的组织也包括进去。

链码容器会在 Org3 的 对等节点第一次调用链码后启动。

升级背书策略

你可以在不重新打包或者重新安装链码的情况下，使用链码定义来更新背书策略。通道成员可以用新的背书策略批准一个链码定义并且提交到链上。

Org1、Org2 和 Org3 同意一个新的背书策略，要求所有三个组织都背书一个交易。它们把序列号从1增加到2，但是不需要更新链码版本。

新的背书策略会在新的定义提交到通道上后奏效。通道成员不需要通过调用链码或者执行 Init 方法来重新启动链码容器，以更新背书策略。

组织把新的链码定义提交到通道上来更新背书策略。

不安装链码的情况下批准定义

通道成员可以在不安装链码包的情况下批准一个链码定义。即使你不想使用链码来背书交易或查询账本，在链码定义被提交到通道之前，你还是可以为链码定义背书。你需要背书的链码定义中不需要包括packageID，其他参数与其他通道成员批准的链码定义相同。

Org3 不需要安装链码包。因此，他们不需要提供 packageID 来作为链码定义的一部分。但是，Org3 仍然可以为已经提交到通道上的标识符为MYCC的链码定义背书。

一个组织不同意链码定义

不批准已经提交到通道上的链码定义的组织不能使用链码。没有批准链码定义或者批准了不同的链码定义的组织，不能在他们的对等节点上执行链码。

Org3 批准了一个和 Org1、Org2 不同的背书策略。结果是，Org3 不能在通道上使用 MYCC 链码。但是，Org1 或者 Org2 仍然能得到足够的背书，可以把该链码定义提交到通道上并且使用链码。提交链码定义的交易仍然能被添加到账本上并且保存到 Org3 的 对等节点上，但是，Org3 不能为交易背书。

一个组织可以批准使用任何序列号或者版本号的链码定义，这样你就可以批准已经提交到通道上的链码定义并且开始使用链码。你还可以批准一个新的链码定义，以纠正在批准或打包链码过程中所犯的任何错误。

通道不同意链码定义

如果通道上的组织不同意一个链码定义，定义就不能被提交到通道上，没有通道成员能用该链码。

Org1、Org2 和 Org3 都批准不同的链码定义。因此，没有链码定义得到了足够的背书，不能把链码定义提交到通道上，没有通道成员能使用链码。

组织安装不同的链码包

每个组织在批准链码定义时，可以使用不同的packageID。这允许通道成员安装不同的链码二进制文件，这些二进制文件使用相同的背书策略，并在相同的链码命名空间中读写数据。

组织可以使用这个功能来安装智能合约，这个智能合约包含其组织特定的业务逻辑。每个组织的智能合约可能包含额外的交易验证，组织需要在对等节点对交易背书之前验证该交易。每个组织还可以编写代码，将智能合约与现有系统中的数据集成在一起。

Org1 和 Org2 分别安装名为 MYCC的链码，该链码包含了它们组织特定的业务逻辑。

用一个包安装多套链码

通过批准和提交多个链码定义，可以使用一个链码包在一个通道上创建多个链码实例。每个实例需要指定一个不同的链码名称。允许在一个通道上运行智能合约的多个实例，但是该合约要服从不同的背书策略。

Org1 和 Org2 用 MYCC_1 链码包来批准和提交两个不同的链码定义。结果，两个对等节点都有两个链码容器。 MYCC1 具备1/2的背书策略（2个组织中需要有1个背书），而 MYCC2 有2/2的背书策略（2个组织都必须背书）。
