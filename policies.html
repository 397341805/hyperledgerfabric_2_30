Policies

策略

​ 受众： 架构师，应用和智能合约开发者，管理员

​ 本主题将包含：

  • 什么是策略
  • 为什么需要策略
  • 如何实现策略
  • 在Fabric中怎么写策略
  • Fabric链码生命周期
  • 重载通道策略

​ 注意：这个话题中将描述一种不使用系统通道（由排序服务启动并且由排序服务独占控制的通道）的网络。自从2.3版本发布以来，与没有创建系统通道的通道相比，使用系统通道被认为是遗留的进程。2.2版本中的这一节介绍的是包含了系统通道的版本。

什么是策略

​ 从根本上来说，策略是一组规则，用来定义如何做出决策和实现特定结果。为此，策略一般描述了谁（who）和什么(what)，比如一个人对资产的访问或者权限。我们可以看到，在我们的日常生活中策略也在保护我们的资产数据，比如汽车租金、健康、我们的房子等。

​ 例如，保险策略定义了保险赔付的条件、条款、限制和到期时间。该策略经过了策略持有者和保险公司的一致同意，定义了各方的权利和责任。保险策略用于风险管理，在 Hyperledger Fabric 中，策略是基础设施的管理机制。Fabric策略表示成员如何就接受或拒绝对网络、通道或智能合约的更改达成一致。策略在网络最初配置的时候由联盟成员一致同意，在网络演化的过程中可以进行修改。例如，他们定义了从通道中添加或者删除成员的标准，改变区块格式或者指定需要给智能合约背书的组织数量。所有这些定义谁可以干什么的行为都在策略中描述。简单来说，你在 Fabric 网络中的所有想做的事情，都要受到策略的控制。

为什么需要策略

​ 策略是使 Hyperledger Fabric 不同于其他区块链系统（比如 Ethereum 或者 Bitcoin）的内容之一。在其他系统中，交易可以在网络中的任意节点生成和验证，管理网络的策略在任何时间点都是固定的，并且只能使用与管理代码的相同流程进行更改。因为 Fabric 是许可链（底层基础设施会对它的用户进行身份验证），所以用户可以在网络启动前决定网络的管理方式，以及改变正在运行的网络的管理方式。

​ 策略决定了那些组织可以访问或者更新 Fabric 网络，并且提供了强制执行这些决策的机制。策略包含对给定资源(如用户或系统链代码)具有访问权的组织的列表。它们还指定了需要多少组织就资源（比如通道或智能合约）更新的提议达成一致。一旦策略被写入，策略就会评估与交易和提案相关的签名集合，并验证签名是否满足网络的管理规则。

Fabric是如何实现策略的

​ 策略是在策略定义的特定操作的相关管理域中定义的。例如，将对等节点组织添加到通道的策略是在对等节点组织（Application group,应用程序组）的管理域中定义的。类似地，将排序节点添加到通道的共识节点集合是由排序节点组（Orderer group）的策略控制的，对排序节点和对等节点的组织域进行的操作包含在通道组（Channel group）中。

​ 通常，这些策略在其所属组中都是默认为“多数人管理”（例如对等节点组织中的大多数对等节点，或者对于通道策略，指的是大多数的对等节点组织和排序节点组织，），但是也可以在用户定义的规则中指定这些策略。

访问控制列表（Access control lists，ACLs）

​ 网络管理员可能对 Fabric 中 ACL 的使用更感兴趣，ACL 通过将资源和已有策略相关联的方式来配置对资源的访问权限。“资源”可以是系统链码中的方法（例如，“qscc”中的“GetBlockByNumber”）或者其他资源（例如，谁可以获取区块事件，Block events）。ACL引用应用程序通道配置中定义的策略，并对它们进行扩展以控制其他资源。Fabric ACL 的默认集合在 configtx.yaml文件的 Application: &ApplicationDefaults部分，但是它们可以也应该在生产环境中被重写。configtx.yaml中定义的资源列表是 Fabric 当前定义的所有内部资源的完整集合。

该文件中，ACL 以如下格式表示：

# ACL policy for chaincode to chaincode invocation
peer/ChaincodeToChaincode: /Channel/Application/Writers

​ 其中，peer/ChaincodeToChaincode表示被保护的资源，/Channel/Application/ writer指的是必须满足的策略，这样相关联的事务才会被认为是有效的。

​ 关于 ACL 更深入的信息，请参考操作指南中的 ACL 主题。

智能合约背书策略

​ 链码包中的每一个智能合约都有一个背书策略，该策略指明了需要通道中多少个不同组织的对等节点根据指定智能合约执行和验证交易才能使一笔交易有效。因此，背书策略定义了必须对提案执行进行“背书”（批准）的组织（通过对等节点）。

修改策略

​ 还有一个对 Fabric 的策略运作起着重要作用的策略类型，即修改策略。修改策略指明了需要对配置更新签名的一组身份。它是定义如何更新策略的策略。因此，每个通道配置元素都包含一个管理其修改的策略的引用。

如何在Fabric中写策略

​ 如果要更改Fabric中的任何内容，与资源关联的策略描述了谁需要批准它，可以通过个人的显式签名，也可以通过一个组的隐式签名。在保险领域，一个显式签名可以是业主保险代理集团中的一员。而一个隐式签名类似于需要业主保险代理集团中的大多数管理成员批准。这很有用，因为这样组中的成员就可以在不更新策略的情况下发生变动。在 Hyperledger Fabric 中，策略中的显式签名使用Signature语法，隐式签名使用ImplicitMeta语法。

签名策略

​ Signature策略定义了一些特定类型的用户，这些用户必须进行签名才能满足策略，例如OR('Org1.peer', 'Org2.peer')。策略是很强大的，因为它可以构造复杂的规则，比如“组织 A 和 2 个其他管理员，或者 6 个组织的管理员中的 5 个”。语法支持AND,OR,NOutOf的任意组合。例如，一个策略可以简单表达为AND('Org1.member', 'Org2.member')，它表示要满足该策略就同时需要 Org1 中的一个成员和 Org2 中的一个成员的签名。

隐元策略

​ 隐元(ImplicitMeta)策略仅在通道配置的上下文中有效，通道配置是基于配置树中的策略的分层层次结构。隐元策略将策略的结果聚合到配置树的更深处，这些配置树最终由签名策略定义。它们是隐式的，因为它们是基于通道配置中的当前组织隐式构造的;它们是元策略，因为它们的评估不依赖于特定 MSP 规范，而是依赖于配置树中它们的其他子策略。

​ 下边的图例说明了应用通道分层的策略结构，并演示了隐元通道配置管理策略（称为 /Channel/Admins）的满足条件，即只有当它的配置层级下面的这些子策略Admins都满足的时候（复选框打勾代表子策略的条件被满足）， /Channel/Admins才能被满足。

​ 正如你在上图看到的隐元策略（Type = 3）使用了一种不同的语法，" "，例如：

`MAJORITY sub policy: Admins`

​ 上面展示了一个子策略Admins，它可以代表配置树下面所有的Admins子策略。你可以创建你自己的子策略并任意命名，并且可以定义在你的每一个组织中。

​ 正如上边提到的，隐元策略比如 MAJORITY Admins（多数人管理） 的主要优势在于当你向通道添加新组织的时候，你不必更新通道策略。因此隐元策略就像联盟成员变更一样灵活。联盟中成员的新增或者退出只要联盟成员一致同意即可，不需要更新策略。重申一下，隐元策略最终处理的是如图所示的配置树中它们之下的签名子策略。

​ 你也可以定义一个应用级别的隐式策略来进行跨组织操作，例如在通道中，需要 ANY （任意）、 ALL （全部）或者 MAJORITY （大多数）组织来满足。这个格式有更好、更自然的默认值，这样组织可以决定有效背书的含义。

​ 如果在组织定义中包含NodeOUs，则可以实现进一步的粒度和控制。组织单元(Organization Units，OU)定义在Fabric CA客户端配置文件中，在创建时可以与身份相关联。在Fabric中，NodeOUs提供了一种在数字证书层次结构中对身份进行分类的方法。例如，一个具备特定的 NodeOUs 的组织需要一个对等节点对它的签名来完成合法的背书，而没有 NodeOUs的组织需要所有能签名的成员来为它进行背书。

一个例子：通道配置策略

​ 理解这些策略要从configtx.yaml开始，它里边定义了通道策略。我们可以使用Fabric测试网络中的configtx.yaml文件来查看这两种策略语法类型的示例。我们将分析 fabric-samples/test-network 示例中使用的configtx.yaml文件。

​ 文件的第一部分定义了将要成为通道成员的组织。在每个组织的定义中设置了默认策略，读者，写者，管理和背书（Readers, Writers, Admins, and Endorsement），不过你也可以任意命名你的策略。每个策略都有一个类型（Type），Type描述了策略的表达式类型（Signature或者ImplicitMeta）以及规则。

​ 下面的测试网络示例展示了通道中组织Org1的定义,

​ 在它的策略中Type是Signature,背书策略的Rule被定义为"OR('Org1MSP.peer')"。此策略指定了必须有一个是Org1MSP成员的对等节点进行签名。这些签名策略正是隐元策略指向的子策略。

- &Org1
       # DefaultOrg defines the organization which is used in the sampleconfig
       # of the fabric.git development environment
       Name: Org1MSP

       # ID to load the MSP definition as
       ID: Org1MSP

       MSPDir: ../organizations/peerOrganizations/org1.example.com/msp

       # Policies defines the set of policies at this level of the config tree
       # For organization policies, their canonical path is usually
       #   /Channel/<Application|Orderer>/<OrgName>/<PolicyName>
       Policies:
           Readers:
               Type: Signature
               Rule: "OR('Org1MSP.admin', 'Org1MSP.peer', 'Org1MSP.client')"
           Writers:
               Type: Signature
               Rule: "OR('Org1MSP.admin', 'Org1MSP.client')"
           Admins:
               Type: Signature
               Rule: "OR('Org1MSP.admin')"
           Endorsement:
               Type: Signature
               Rule: "OR('Org1MSP.peer')"

​ 下一个示例展示了configtx.yaml的应用程序部分中使用的隐元策略类型。这些策略集位于路径/Channel/Application/上。如果使用缺省的Fabric ACL集，这些策略将定义应用程序通道的许多重要的行为，例如谁可以查询通道账本、调用链码或更新通道配置。这些策略指向了每个组织定义的子策略。之前Org1部分中包含的子策略Reader, Writer, Admin将被应用程序部分中的隐元策略Reader, Writer, Admin评估。因为测试网络是用默认策略构建的，所以您可以使用Org1来查询通道帐本、调用链码，或者批准您创建的通道的通道更新

################################################################################
#
#   SECTION: Application
#
#   - This section defines the values to encode into a config transaction or
#   genesis block for application related parameters
#
################################################################################
Application: &ApplicationDefaults

    # Organizations is the list of orgs which are defined as participants on
    # the application side of the network
    Organizations:

    # Policies defines the set of policies at this level of the config tree
    # For Application policies, their canonical path is
    #   /Channel/Application/<PolicyName>
    Policies:
        Readers:
            Type: ImplicitMeta
            Rule: "ANY Readers"
        Writers:
            Type: ImplicitMeta
            Rule: "ANY Writers"
        Admins:
            Type: ImplicitMeta
            Rule: "MAJORITY Admins"
        LifecycleEndorsement:
            Type: ImplicitMeta
            Rule: "MAJORITY Endorsement"
        Endorsement:
            Type: ImplicitMeta
            Rule: "MAJORITY Endorsement"

Fabric链码生命周期

​ Fabric 2.0 发布版本中，介绍了一个新的链码生命周期过程，其中使用了一种更民主的过程来管理网络中的链码。这个过程允许多个组织在链码应用到通道之前对它的操作方式进行投票。这很重要，因为这是这个新的生命周期过程和在该过程中指定的策略的组合，它们决定了整个网络的安全性。该流程的更多细节在 Fabric chaincode lifecycle话题中，但是为了本话题的目的，你应该理解策略在流程中是如何被使用的。

​ configtx.yaml文件中Application部分包含了默认的链码生命周期背书策略。在生产环境中你应该为你的用例自定义这个。

​ 

################################################################################
#
#   SECTION: Application
#
#   - This section defines the values to encode into a config transaction or
#   genesis block for application related parameters
#
################################################################################
Application: &ApplicationDefaults

    # Organizations is the list of orgs which are defined as participants on
    # the application side of the network
    Organizations:

    # Policies defines the set of policies at this level of the config tree
    # For Application policies, their canonical path is
    #   /Channel/Application/<PolicyName>
    Policies:
        Readers:
            Type: ImplicitMeta
            Rule: "ANY Readers"
        Writers:
            Type: ImplicitMeta
            Rule: "ANY Writers"
        Admins:
            Type: ImplicitMeta
            Rule: "MAJORITY Admins"
        LifecycleEndorsement:
            Type: ImplicitMeta
            Rule: "MAJORITY Endorsement"
        Endorsement:
            Type: ImplicitMeta
            Rule: "MAJORITY Endorsement"

​ LifecycleEndorsement策略规定了谁需要对链码定义进行批准。

​ Endorsement策略是链码默认的背书策略。下面有更多的介绍。

链码背书策略

​ 当使用 Fabric 链码生命周期的链码被批准并提交到通道时会为其指定一个背书策略（这个背书策略会覆盖与该链码相关的所有状态）。背书策略可以通过引用通道配置中定义的背书策略来指定，或者显式地指定签名策略。

​ 如果在批准阶段没有明确指明背书策略，就默认使用Endorsement策略"MAJORITY Endorsement"，意味着要想使交易生效就需要大多数不同通道成员（组织）去执行并验证交易。默认策略允许加入通道的组织自动加入链码背书策略。如果你不想使用默认背书策略，你可以使用签名策略格式来指定更复杂的背书策略（这样就需要链码先被通道中的一个组织签名，然后让其他组织签名）。

​ 签名策略还允许包含principals，它是将标识与角色匹配的一种方式。principals类似于用户ID或组ID，但是它的用途更为广泛，因为它们可以包含参与者标识的各种属性，例如参与者的组织、组织单元、角色，甚至参与者的特定标识。我们讨论的principals是决定其权限的属性。principals的描述方式是'MSP.ROLE'，其中MSP代表MSP ID（组织），ROLE代表四种角色之一：成员，管理员，客户，对等节点（Member, Admin, Client, and Peer）。当用户注册到CA时，角色就会与身份相关联。您可以自定义Fabric CA上可用的角色列表。

​ 下面是一些合法的principals的例子：

  • ‘Org0.Admin’: Org0 MSP 的一个管理员
  • ‘Org1.Member’: Org1 MSP 的一个成员
  • ‘Org1.Client’: Org1 MSP 的一个客户端
  • ‘Org1.Peer’: Org1 MSP 的一个对等节点
  • ‘OrdererOrg.Orderer’: OrdererOrg MSP 的一个排序节点

​ 在某些情况下，特定的状态(换句话说，特定的键值对)可能需要有不同的背书策略。这种基于状态的背书允许特定的键值使用不同的背书策略来重载默认的链码级别背书策略。

​ 要了解更多关于如何写一个背书策略请参考话题 Endorsement policies。

​ 注意，不同版本 Fabric 的策略有所不同：

​ 在 Fabric 2.0 之前，链码的背书策略可以在链码实例化或者使用链码生命周期指令时更新。如果在实例化时没有指定，那么背书策略默认为“通道中组织的任何成员”。例如，包含组织“Org1”和“Org2”的通道将具有默认的背书策略“OR(‘Org1.member’, ‘Org2.member’)”。

​ 从Fabric 2.0开始，Fabric引入了一个新的链码生命周期过程，允许多个组织在链码被应用到通道上之前就如何操作达成一致。新的过程要求各个组织对定义链码的参数，如名称、版本和链码背书策略达成一致。

重载策略定义

​ Hyperledger Fabric包括默认策略，这些策略对于入门、开发和测试区块链非常有用，但它们需要在生产环境中被自定义。你应该知道configtx.yaml文件中的默认策略。在configtx.yaml中除了默认的Readers, Writers, Admins以外，你还可以用任意的动词进行扩展。

​ 要了解更多关于在创建通道时重载策略定义的信息，可以参考Channel policies 和Creating a channel without a system channel。

​ 要了解更多关于如何更新通道，可以参考Updating a channel configuration。
